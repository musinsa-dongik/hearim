# 헤아림 데일리 스킬

"내가 안다고 생각한 것을, 말로 헤아려보는 시간"

## 사전 준비

### 사용자 설정 파일

`~/.hearim-config`에 사용자별 설정을 둔다. 없으면 기본값을 사용한다.

```bash
# ~/.hearim-config 예시
HEARIM_REPOS=(
  "$HOME/company/github/global/global-musinsa-frontend"
  "$HOME/company/github/global/global-ui"
  "$HOME/hearim"
)
```

설정 파일이 없으면 현재 레포의 커밋만 수집한다.

---

## 데일리 모드

### Step 1. 컨텍스트 수집 (병렬 실행)

아래 소스를 **모두 빠짐없이** 동시에 조회한다.
하나라도 누락하면 학습 맥락이 불완전해지므로, 반드시 전체 소스를 수집한다.

---

#### 1-1. Git 커밋 (모든 프로젝트)

헤아림은 **업무 중 학습한 내용**을 기록하므로, 모든 프로젝트의 커밋을 수집해야 한다.

##### 현재 레포
```bash
GIT_EMAIL=$(git config user.email)
git log --since="오늘 00:00" --until="내일 00:00" --author="$GIT_EMAIL" --all --format="%h %s (%ai)" --stat
```

##### 다른 프로젝트 레포
`~/.hearim-config` 파일이 있으면 거기에 정의된 레포들을 조회한다:
```bash
GIT_EMAIL=$(git config user.email)
if [ -f ~/.hearim-config ]; then
  source ~/.hearim-config
  for repo in "${HEARIM_REPOS[@]}"; do
    echo "=== $(basename $repo) ==="
    git -C "$repo" log --since="오늘 00:00" --until="내일 00:00" --author="$GIT_EMAIL" --all --format="%h %s (%ai)" --stat 2>/dev/null
  done
fi
```

##### Fallback (커밋이 0개일 때)
현재 레포에서 최근 5개 커밋을 조회하여 마지막 작업일 이후 맥락을 파악한다:
```bash
git log --author="$GIT_EMAIL" --all --format="%h %s (%ai)" --stat -5
```

---

#### 1-2. MEMORY.md (여러 경로 조회)

반드시 아래 경로를 조회한다. 하나라도 빠지면 작업 맥락을 놓친다.

##### Auto Memory 디렉토리
```
Glob: ~/.claude/projects/**/memory/MEMORY.md
```
발견된 파일 중 **최근 수정된 상위 5개**를 읽어 학습 맥락을 파악한다.

---

#### 1-3. 이전 헤아림 데일리

학습의 연속성을 위해 가장 최근 헤아림 데일리를 읽는다:
```
Glob: ~/ai-workspace/hearim/daily/*.md → 최신 파일 읽기
```
> 이전 엔트리의 **추가 학습 질의**와 **배움 카드 주제**를 확인하여 오늘의 학습과 연결한다.

---

### Step 2. 학습 포인트 추출

수집한 데이터에서 **배움 카드로 만들 주제**를 식별한다:

1. **Git 커밋 분석**: 커밋 메시지와 변경 파일에서 새로운 패턴, 기술적 결정 추출
2. **MEMORY.md 분석**: 설계 결정사항, 코드 리뷰 피드백, 아키텍처 패턴에서 학습 포인트 추출
3. **이전 헤아림 연결**: 이전 "추가 학습 질의"에 대한 답을 오늘 찾았는지 확인
4. **반복 패턴 식별**: 여러 소스에서 반복적으로 나타나는 주제 = 핵심 학습 포인트

### Step 3. 데일리 작성

아래 템플릿을 따라 작성한다. 섹션 순서를 지킨다.

```markdown
## 헤아림 데일리: YYYY-MM-DD (요일)

### 오늘의 작업 (Context)

- **이슈:** [무엇이 문제였는지 / 무엇을 했는지]
- **원인:** [왜 발생했는지 / 왜 이 작업이 필요했는지]
- **해결:** [어떻게 해결했는지 / 어떤 접근을 택했는지]

> 작업이 여러 건이면 각각 이슈/원인/해결을 반복한다. 작업 간에는 빈 줄로 구분한다.
> 코드 리뷰, 커뮤니케이션 등 비개발 작업도 포함한다.

---

### 배움 카드 (Concepts)

#### 1. [개념/패턴 이름]

[3-5문장으로 설명. "무엇을", "왜", "어떻게"를 포함]

**패턴/문법:**
- [코드 예시나 패턴 설명]

**언제 쓰는가:**
- [적용 상황 설명]

---

#### 2. [개념/패턴 이름]

[동일 구조로 작성]

---

### 추가 학습 질의

- [ ] [더 깊이 탐구할 주제 1]
- [ ] [연관된 학습 영역 2]

---

### 튜터의 한마디

[메타인지적 관점에서 오늘의 학습을 정리. 성장 방향 제시. 2-3문장.]
```

### 작성 기준

- 배움 카드: 최소 2개, 최대 4개
- 각 카드는 1줄 요약이 아닌 3-5문장 설명
- "헷갈린다", "추가 학습 필요" 표현 허용
- 과장 없이 간결하고 구체적으로

### Step 4. 로컬 파일 저장

```
출력 경로: ~/ai-workspace/hearim/daily/YYYY-MM-DD.md
```

파일이 이미 존재하면 내용을 덮어쓴다.

### Step 5. 완료 보고

사용자에게 알린다:
- 저장된 파일 경로
- "DB 저장은 `/hearim-push`로 별도 실행하세요."

---

### Step 6. 학습 연결

- 이전 헤아림 엔트리의 "추가 학습 질의"와 오늘의 배움을 연결
- 반복되는 테마나 지식 갭 식별
- 성장 궤적 확인 (이전에 "헷갈린다"고 했던 것이 오늘 해결되었는지)

---

## 튜터 페르소나

모든 작성 시 아래 태도를 따른다:
- 친절한 선임 개발자 톤
- 선(先) 설명, 후(後) 심화
- 핵심 포인트에서 전문적 식견 제시
- 설명이 막히는 지점 = 성장 포인트로 인식

---

## 수집 체크리스트

데일리 작성 전 아래 항목이 모두 수집되었는지 확인한다:

- [ ] Git 커밋: 현재 레포 + 다른 프로젝트 레포 (전체 조회)
- [ ] MEMORY.md: auto memory 디렉토리의 최근 수정 파일
- [ ] 이전 헤아림 데일리: 최근 파일의 추가 학습 질의 확인

---

## 주의사항
- 민감한 정보(API 키, 비밀번호, 내부 URL 등)는 제외한다
- content에 이모지를 쓰지 않는다 (섹션 제목 포함)
- 헤아림 스터디 발표 자료로 활용할 수 있도록 "설명하듯이" 정리한다
